  <action>
Create `scripts/rebuild_orchestrator.py` -- an async CLI script using httpx and asyncio.

**Structure:**

1. INSTRUCTIONS list: A predefined list of dicts, each with:
   - `label`: Short human-readable name (e.g., "Add status field to Document")
   - `instruction`: Full natural language instruction for the agent
   - `expected_files`: List of files expected to be modified (for verification logging)

   Include 4-5 graduated instructions matching the CONTEXT.md spec:
   - Simple: "Add a 'status' field of type string to the Document interface in src/types/index.ts. It should accept values 'draft', 'published', or 'archived'."
   - Simple: "Add a GET /ready endpoint to src/routes/health.ts that returns { status: 'ready', uptime: process.uptime() }."
   - Medium: "Add a new route file src/routes/documents.ts with GET /documents (returns all documents) and GET /documents/:id (returns one document or 404). Use the findAll and findById functions from src/models/document.ts. Register the router in the main app."
   - Complex: "Add a 'tags' field (string array) to the Document interface, update the create function in src/models/document.ts to accept tags, and add a DocumentList filter that only shows documents matching a selected tag."
   - Multi-agent: "In parallel: (1) Add a 'priority' field to Document interface with values 'low', 'medium', 'high' and (2) Add a DELETE /documents/:id endpoint to src/routes/documents.ts."

2. `send_instruction(client, host, instruction, working_dir, project_id)` async function:
   - POSTs to /instruction with the instruction text, working_directory, and project_id
   - Returns run_id

3. `poll_status(client, host, run_id, timeout=300, interval=3)` async function:
   - Polls GET /status/{run_id} every `interval` seconds
   - Returns final status dict when status is terminal (completed, failed, error, cancelled)
   - Raises TimeoutError after `timeout` seconds

4. `log_result(log_path, index, label, instruction, status, duration, trace_url)` function:
   - Reads SHIP-REBUILD-LOG.md
   - Appends a row to the Instructions Log table: | {index} | {label} | {status} | {duration}s | {needs_intervention} | {trace_url or ""} |
   - If status is "failed" or "error", also appends a template entry to the Interventions section with the instruction number and error details pre-filled, "What Was Done Manually" and "What It Reveals" left as TODO placeholders for the user

5. `run_rebuild(host, working_dir, project_id, dry_run, start_from)` async main function:
   - Iterates over INSTRUCTIONS starting from `start_from` index
   - For each: prints instruction, sends it (or skips if dry_run), polls for result, logs outcome
   - Tracks summary stats: total, succeeded, failed, interventions needed
   - Prints final summary to stdout

6. CLI via argparse:
   - `--host` (default: http://localhost:8000)
   - `--working-dir` (default: absolute path to ship-rebuild/)
   - `--project-id` (default: None, will use "default")
   - `--dry-run` (preview instructions without sending)
   - `--start-from` (int, default 1, resume from instruction N)
   - `--log-path` (default: SHIP-REBUILD-LOG.md)
   - `--timeout` (default: 300 seconds per instruction)

Include module docstring: "Orchestrates the Ship rebuild by sending graduated instructions to the Shipyard agent."
Include shebang and main guard.
Use httpx.AsyncClient for HTTP calls.
Handle connection errors gracefully with retry (3 attempts, 5s backoff).
