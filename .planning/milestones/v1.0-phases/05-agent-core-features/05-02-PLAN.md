1. Add a new routing function `after_reporter`:
```python
def after_reporter(state: dict) -> str:
    """Route to git_ops if edits were made, otherwise end."""
    edit_history = state.get("edit_history", [])
    if edit_history:
        return "git_ops"
    return "end"
```

2. Replace `graph.add_edge("reporter", END)` with:
```python
graph.add_conditional_edges("reporter", after_reporter, {
    "git_ops": "git_ops",
    "end": END,
})
```

3. Keep the existing `graph.add_edge("git_ops", "reporter")` -- wait, this would create a cycle: reporter -> git_ops -> reporter. Instead, change git_ops to go to END:
   - Remove `graph.add_edge("git_ops", "reporter")` (line 176)
   - Add `graph.add_edge("git_ops", END)`
   - The classify_step "git_ops" route should now also go to END after git_ops finishes. Since we already have git_ops -> END now (replacing git_ops -> reporter), when classify_step routes to git_ops for a plan step of kind "git", it will go to END after. But we need it to continue to the next step. So instead:

   Better approach: git_ops after reporter should go to END. git_ops from classify_step should go back to should_continue (advance). But they're the same node. So use a flag.

   Simplest approach: Add a new node `auto_git_ops` that wraps git_ops_node and always goes to END. Keep the existing git_ops node for plan-step triggered git:

   Actually, the cleanest approach per the research: change the graph wiring so:
   - `git_ops` (from classify_step) -> `advance` -> `classify` (continues the plan loop)
   - `reporter` -> conditional -> `auto_git` -> END

   Add a new node `auto_git` that calls git_ops_node:
   ```python

   ```

   Replace `graph.add_edge("reporter", END)` with:
